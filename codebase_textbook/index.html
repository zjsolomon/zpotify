Inside zpotify

Building a Terminal Music Client From Scratch — Terminals, Streams, and Real-Time Audio in Python

About this book

zpotify is a Spotify client that runs entirely in a terminal: no curses, no HTTP library, no audio framework beyond a thin PortAudio binding. Roughly 5,400 lines of Python across 31 modules. That constraint — write it yourself — makes it an unusually good specimen to study, because every layer that a framework would normally hide is here in readable form: the escape-code renderer, the OAuth flow, the ring buffer, the real-time mixer.

This book assumes you are comfortable with Python and have met threads before. It does not assume you know anything about termios, OAuth, or digital audio. Each chapter ends with exercises; solutions are in the appendix.

Chapter 1The Shape of the Problem

Every interesting design decision in zpotify follows from one fact: Spotify does not let you download audio through its Web API. The API will tell you what is playing, search the catalogue, and control playback on a device you own. It will never hand you a single PCM sample.

So the program is split down the middle. Two independent channels reach Spotify, and they never carry each other's traffic:

ChannelCarriesImplemented by
Control search, playlists, play/pause/seek, queue, what's playing api.py + auth.py, stdlib urllib only
Audio raw PCM samples librespot, an external Rust binary, as a subprocess

librespot is an open-source reimplementation of the Spotify client protocol. Run in pipe mode, it logs into your account, registers itself on the network as a Spotify Connect device named zpotify, and writes decoded audio to its standard output as raw signed 16-bit little-endian stereo frames at 44,100 Hz. Nothing else. No control surface, no UI. The codebase describes it in one phrase worth remembering: librespot is used strictly as an audio faucet.

That yields the whole architecture. To play a song, zpotify does not send audio anywhere — it asks the Web API to start playback on the librespot device, and then reads the bytes that come out of the subprocess's stdout. The two channels meet only in the user's ears.

  +------------+    Web API (urllib)     +---------+
  |  zpotify   | ----------------------> | Spotify |
  |            | <---------------------- |         |
  +------------+    playback state       +---------+
        ^                                      |
        | stdout: S16LE 44100 stereo           | Connect
        |                                      v
  +-----+------+                         +-----------+
  | ring buffer| <---------------------- | librespot |
  +-----+------+                         +-----------+
        | sounddevice callback
        v
     speakers

What this costs

Three consequences fall out immediately, and you will meet all of them again later.

Premium is mandatory. Spotify only permits playback outside its official clients for Premium accounts — a licensing rule, not a technical one, and nothing in the code can route around it.

State is remote and laggy. zpotify does not know what is playing; it polls /me/player every two seconds (POLL_INTERVAL in ui/app.py). Your phone can change the track and zpotify finds out up to two seconds later. Chapter 6 shows the optimistic-UI machinery that hides this lag, and Chapter 5 shows what happens when the lag collides with real-time audio.

The API is a hostile dependency. Spotify has been steadily restricting what development-mode applications may call. Endpoints do not merely fail — they fail inconsistently. Comments throughout api.py read like field notes from a long campaign: search limits above 10 are rejected; the documented /playlists/{id}/tracks returns 403 while /items works and keys its wrapper item instead of track; dev-mode servers throw 5xx in bursts.

A lesson worth extracting

A 200 response does not mean the data is there. /artists/{id} answers happily but returns an object stripped of its genres and popularity fields. Code written against the documentation — or against a status code — will look correct and silently do nothing. Probe payloads, not statuses.

Chapter 2The Terminal Engine

A terminal is a byte stream pretending to be a screen. You write escape sequences into it; you read escape sequences out of it. term/ is the layer that makes that bearable, and it is written from scratch — the module docstring is blunt about it: "No curses."

Raw mode and the alternate screen

Normally the kernel's line discipline buffers input until Enter, echoes characters, and turns Ctrl-C into a signal. An interactive full-screen program wants none of that. Screen.enter() saves the current termios settings, switches the terminal to raw mode, and enables three things by escape code: the alternate screen (so the user's scrollback survives), mouse reporting, and bracketed paste. exit() reverses all of it and is also wired to atexit, because a program that crashes while owning a raw tty leaves the user with an unusable shell.

The frame buffer and the differ

Redrawing an entire terminal 30 times a second would produce visible tearing and waste enormous bandwidth over SSH. So Screen keeps two frames of cells, where a cell is a (char, Style) pair:

Cell = tuple[str, Style]
_NULL_CELL: Cell = ("\uffff", DEFAULT)   # unmatchable sentinel

Drawing calls (put, fill, box) mutate only the current frame. present() then walks both frames, finds runs of cells that differ, and emits just those runs:

for y in range(self.rows):
    cur, prev = self._cur[y], self._prev[y]
    x = 0
    while x < self.cols:
        if cur[x] == prev[x]:
            x += 1; continue
        run_start = x
        while x < self.cols and cur[x] != prev[x]:
            x += 1
        pen = self._emit_run(out, y, run_start, x, cur, pen)
self._write("".join(out)); self._flush()
self._prev = [row[:] for row in self._cur]

Two details in that loop carry more weight than they look. The output is accumulated in a list and written once — a single write plus a single flush per frame, so the terminal never renders a half-updated screen. And pen threads through _emit_run, tracking the style currently active in the terminal so a style escape is emitted only when the style actually changes.

The _NULL_CELL sentinel is the trick that makes resizing correct. Fill the previous frame with a character that can never legitimately appear, and every cell compares unequal, so the next present() repaints everything. One sentinel replaces an entire "force full redraw" code path.

Colors that survive contact with real terminals

Styles are authored as 24-bit RGB. Whether they can be sent that way is a different question, resolved once at startup by detect_color_mode() (term/style.py) in strict precedence:

ZPOTIFY_COLOR override -> TERM_PROGRAM == Apple_Terminal -> COLORTERM -> "256"

Apple Terminal is special-cased because it advertises truecolor support and then mangles 38;2 sequences — so it is forced down the indexed path regardless of what it claims. When truecolor is unavailable, xterm256() quantizes each color, considering both the 6×6×6 color cube and the 24-step greyscale ramp and choosing whichever is nearer in squared distance. Greys matter: a dimmed interface hits the greyscale ramp constantly, and cube-only quantization would tint it.

Reading input: the escape problem

term/input.py parses the byte stream back into Key, Mouse, Paste, and Resize events. Most of it is table-driven decoding of CSI and SS3 sequences. One case is genuinely undecidable.

When you press Escape, the terminal sends 0x1b. When you press the up arrow, it sends 0x1b [ A. On seeing 0x1b with nothing after it, the parser cannot know whether the user pressed Escape or whether the rest of an arrow-key sequence is still in flight. There is no correct answer available from the bytes alone — only from time. So the reader exposes pending_escape, and the main loop shortens its select timeout to ESC_TIMEOUT = 0.025 seconds; if nothing arrives in that window, flush_escape() resolves it as a real Escape.

Design note

25 ms is a deliberate compromise. Too short and a slow SSH link splits an arrow-key sequence into a spurious Escape; too long and Escape feels sluggish. This is the only place in the codebase where correctness is bought with a timing guess — worth noticing, because Chapter 5 is about a subsystem that refuses to make that trade.

Chapter 3Talking to Spotify

Both modules in this chapter are written against urllib and http.server. No requests, no OAuth library.

PKCE, and why a terminal app needs it

The classic OAuth authorization-code flow proves the application's identity with a client secret. A program distributed to users cannot hold a secret — anyone can read it out of the package. PKCE (Proof Key for Code Exchange) replaces the secret with a value invented fresh for each login:

verifier  = secrets.token_urlsafe(64)[:64]           # random, kept in memory
challenge = base64url(sha256(verifier))              # sent to Spotify up front

zpotify sends only the challenge when opening the browser. Spotify redirects back with an authorization code, and only then does zpotify send the verifier. Spotify hashes it and checks it matches the challenge it stored. An attacker who intercepts the redirect gets a code they cannot exchange, because they never saw the verifier. Nothing secret was ever shipped.

The redirect needs somewhere to land, so login_interactive() starts a one-shot http.server on 127.0.0.1:8898, opens the browser, waits for the callback, serves a small styled confirmation page, and shuts down.

Token lifecycle

Tokens live in ~/.config/zpotify/tokens.json, written chmod 600. Access tokens expire, so get_access_token() refreshes them — and it does so EXPIRY_SLACK_SECONDS = 60 early, so a token never expires midway through a request that was valid when it started.

That method is called from every worker thread in the program, so it holds a lock. Two threads noticing expiry simultaneously must not both burn the refresh token.

The API client as an exercise in defensive programming

SpotifyAPI._request() is where the hostility of Chapter 1 is absorbed. Its retry ladder is worth reading closely, because each rung encodes a real failure:

StatusResponseWhy
401force a token refresh, retry once a cache that looked fresh but wasn't; once, so a genuine auth failure surfaces
429sleep for Retry-After (capped), retry rate limiting; the cap stops a hostile header hanging the app
5xxtwo retries, 0.8 s then 2.0 s dev-mode servers fail in bursts that ride out

Beyond the ladder sit smaller scars: gzip bodies that urllib did not decode are detected by magic number and inflated by hand; player commands that return HTTP 200 with unparseable bodies are treated as successes, because the command did succeed; search results are paged ten at a time and de-duplicated, because Spotify's pages overlap.

Chapter 4The Audio Path

This is where the program stops being a network client and becomes a real-time system. Three threads touch audio, and they must never block each other:

  1. a reader thread draining librespot's stdout;
  2. the audio callback, called by PortAudio on its own high-priority thread, which must return within milliseconds or the speakers click;
  3. the UI thread, which only ever reads.

A ring buffer addressed by absolute positions

The buffer between reader and callback is a fixed numpy array, but it is not indexed directly. Two monotonically increasing counters describe it:

self._written = 0   # absolute frames ever received from librespot
self._readpos = 0   # absolute frames ever consumed by the callback

Physical slots come from position % capacity. Absolute positions never wrap, which means the difference between any two of them is meaningful arithmetic: _written - _readpos is exactly how much audio is buffered, and — crucially for Chapter 5 — a track boundary can be recorded as a plain integer position and compared against the playback cursor later, with no modular reasoning at all.

Backpressure is the pacing mechanism

Here is the single most important idea in the audio layer. The writer blocks:

free = self._fill_limit - self._buffered
if free <= 0:
    self._cond.wait(timeout=0.1)
    continue

When the ring is full, the reader thread stops reading. The OS pipe between librespot and zpotify fills. librespot's own write blocks. The entire chain stalls — and it un-stalls only when the audio callback consumes frames, which happens in real time because that is what a sound card does.

Nothing anywhere schedules playback. There is no timer, no sleep, no rate calculation. The speakers pull, and everything upstream is pulled with them.

Why the limit is small

_base_fill is about 0.3 seconds. A large buffer would be safer against underruns but would make every control feel unresponsive: audio already in the buffer has to be discarded or heard. The size of this buffer is a direct trade between robustness and responsiveness — and Chapter 5 changes it, with consequences.

Partial reads and frame alignment

The pipe is unbuffered, so a read can return a byte count that is not a whole number of frames — a frame being four bytes (two channels × int16). Truncating that remainder would shift every subsequent sample by a byte or two, swapping channels and turning music into static. So the loop carries it:

data = pending + data
usable = len(data) - (len(data) % frame_bytes)
pending = data[usable:]

The callback

PortAudio calls _callback with a block to fill. It gates on a small prebuffer (_primed) so a fresh start does not stutter; on underrun it pads with silence and re-arms that gate; it applies local volume and a fade envelope; and it wraps everything in a bare except, because an exception escaping an audio callback takes the stream down permanently.

It also records what it played into a small tap buffer. That tap — post-mix, pre-volume — is what SpectrumAnalyzer reads, which is why the visualizer responds to what you actually hear rather than what was requested. The analyzer does an FFT into 48 log-spaced bins from 40 Hz to 16 kHz with fast attack, slow decay, and slow-falling peak holds.

Chapter 5Crossfade: The Hard Part

Crossfade blends the end of one track into the start of the next. It is optional, off by default, and it is by a wide margin the most demanding code in the project — because it forces three clocks into agreement: Spotify's idea of the current track, librespot's byte stream, and the sound card's playback position.

Finding the boundary

To mix the tail of track N with the head of track N+1, you must know exactly where N ends in the byte stream. The audio is an undifferentiated PCM stream; nothing in it marks a track change.

The naive approach is to detect the pause while librespot loads the next track. The codebase tried it and abandoned it: fast cached loads produce no measurable stall at all. The working approach uses librespot's --onevent hook, which fires an end_of_track event — and, decisively, fires it only on natural completion, never on a skip or a seek. The event is the definition of "the track ended by itself".

Converting that event into a stream position is an accounting problem, because the event arrives while audio is in flight in three separate places:

boundary = self._written              # frames already in the ring
         + self._write_backlog        # frames the blocked writer holds
         + self._pipe_frames()        # frames still in the OS pipe (FIONREAD)
         - lag_frames                 # staleness of the observation

_pipe_frames() issues a FIONREAD ioctl to ask the kernel how many bytes are sitting unread in the pipe. Omit that term and the boundary lands early by however much the pipe happened to hold, and the crossfade begins before the track ends.

Mixing

Boundaries are queued. As playback approaches one, the consumer mixes ring[cursor] (outgoing tail, faded out) with ring[cursor + X] (incoming head, faded in) for X frames, then jumps the cursor past the head region it has already played. The fade is equal-power — gains follow a curve whose squares sum to one — because linear fades dip in perceived loudness at the midpoint, where both signals are at half amplitude.

This is also why crossfade needs a bigger buffer. The mixer must hold the tail of one track and the head of the next simultaneously, so enabling it widens the fill limit to 2X + base.

Three clocks, one screen

Now the interesting failure. With a 5-second crossfade, Spotify reports the new track roughly 5 seconds before you can hear it. Naively adopting the poll would flip the title while the previous song is still playing.

So the UI parks track changes. When a poll reports a new track while a boundary is still in flight, app.py stores it as (state, seen_at, boundaries_played) and keeps displaying the outgoing track. Each tick then asks whether the parked change has become real:

ConditionAction
seen_at < _action_atdiscard — the user acted; the snapshot is stale
boundaries_played > played_beforeadopt — the blend is audible
no boundary, 3 s elapsedflush and adopt — it was a remote skip after all
12 s + fade elapsedflush and adopt — failsafe

Note the counter comparison. It asks whether this specific parked change became audible, not merely whether some boundary played — immune to an earlier boundary satisfying a later park. And note the failsafe: a mark that never plays must never be able to freeze the interface. In a subsystem coordinating three clocks, "this cannot happen" is not a safety argument.

A live bug — read this before Exercise 4

Manual track changes call audio.flush(), which drops the whole buffer, from the API success callback — one network round trip after the keypress. With crossfade on, the fill limit is 2X + base: 10.3 s at X=5. In that window librespot has already written seconds of the new track, and the flush discards them along with the old track's tail, so playback resumes several seconds in. With crossfade off the same code discards 0.3 s and nobody notices. The bug scales exactly with a buffer size that exists for an unrelated feature.

Chapter 6The Event Loop

One thread owns the terminal and all UI state. Everything else reports to it.

Selectors and the self-pipe trick

The loop must wake for two unrelated things: keyboard input, and background work finishing. The first is a file descriptor; the second is not — a queue.Queue has no fd, so select cannot wait on it.

The fix is old and elegant. WorkerPool creates a pipe, registers the read end with the selector, and writes one byte whenever a job completes. Now "a result is ready" is a readable descriptor:

sel.register(self.input, selectors.EVENT_READ)
sel.register(self.workers, selectors.EVENT_READ)   # fileno() -> pipe read end
sel.register(self.control, selectors.EVENT_READ)

Results are queued, not executed, by the worker. drain() runs the callbacks on the UI thread — so callbacks may touch UI state freely, and no widget needs a lock.

Two pools, deliberately

There are two pools, and the reason is ordering. The general pool has three threads for searches and library loads, where concurrency is a win. Player commands go to a single-threaded pool, because pause, play and seek must reach Spotify in the order the user pressed them. Three threads racing to deliver "pause" and "next" would occasionally deliver them backwards.

Optimistic UI and the staleness rule

A two-second poll would make every keypress feel dead, so local actions mutate state immediately: press pause and the icon flips at once, before Spotify is told. This creates a race — a poll already in flight will return the old state and undo the optimistic update.

The rule is one timestamp. _mark_action() records _action_at; every poll remembers when it was requested; and any response whose request predates the last action is dropped:

if requested_at < self._action_at:
    return   # snapshot predates a local action: stale

Views

Seven tabs implement a four-method interface: on_show for lazy loading, render, handle_key returning whether it consumed the key, and a wants_text flag that suppresses single-character hotkeys while a text field has focus. Global keys are dispatched first; anything unclaimed falls through to the active view.

Clickable regions are registered during render via app.add_hit(x, y, w, h, handler) and cleared before every frame. Hit regions cannot drift out of sync with what is drawn, because they are produced by the same code that draws it.

Chapter 7Living With the Codebase

Configuration

Everything persistent is under ~/.config/zpotify/ — overridable with ZPOTIFY_CONFIG_DIR, which is what the test suite uses to avoid touching real credentials. Adding a setting means touching exactly three places: a field on Config, an entry in _KNOWN_KEYS, and a Setting in ui/views/settings.py. Miss the second and your setting silently fails to persist.

Themes without a repaint

Theme styles are plain module attributes in ui/theme.py, read at render time rather than captured at import:

screen.put(x, y, text, theme.DIM)     # looked up each frame
theme.apply("cyan")                   # rebinds the module attributes

Because every widget re-reads them on every frame, theme.apply() re-skins the entire interface live. There is no observer, no invalidation, no repaint call — the 30 Hz render loop already re-reads everything.

Testing a program made of side effects

179 tests run in under a second with no network and no terminal. That is possible because the awkward dependencies are injectable: Screen accepts an output stream and a fixed size, so rendering is tested against io.StringIO; AudioEngine accepts an in-memory stream; WorkerPool.submit is monkeypatched to capture jobs so asynchronous code can be stepped synchronously.

The suite's centre of gravity is where the reasoning is hardest — parked adoption, boundary arithmetic, escape-sequence decoding — rather than spread evenly. That is the right instinct: test where you are most likely to be wrong.

A cautionary tale from this repository's history

The crossfade wiring in app.py was once destroyed by a stray git checkout while uncommitted. It was reconstructible only because the test suite specified its behavior precisely enough to rebuild against. Anything the tests did not pin — a dynamic buffer limit, among other things — was lost silently. Tests are a specification you can execute; untested behavior is folklore.

Release

Publishing is tokenless. Pushing a v* tag triggers a GitHub Actions workflow that runs the tests and then publishes to PyPI via trusted publishing (OIDC), so no credential exists to leak. CI refuses to publish if the tag disagrees with the version in pyproject.toml.

AppendixSolutions to Exercises

Chapter 1

1. The Web API exposes no audio endpoint at all — by design, for licensing reasons. It can only direct playback to a device already authenticated on the account. librespot exists to be such a device while handing the samples to us.

2. Replace the audio channel only: player/librespot.py and the source feeding the ring buffer. Untouched: term/ entirely, the ring buffer and callback, the FFT, the event loop, the views' rendering. The control channel (api.py, auth.py) would go too, since local files need no Spotify account — which is exactly why the two-channel split is the right seam.

3. Press pause and nothing happens for up to two seconds. The mitigation is optimistic UI: mutate local state immediately and reconcile later (Chapter 6).

Chapter 2

1. Assignment would alias both names to the same list objects; the next frame's drawing would mutate _prev too, every cell would compare equal, and the screen would freeze while the program appeared to run.

2. The wide glyph already occupies two columns, so printing a second character for the continuation cell would push the rest of the line one column right and corrupt the layout — and the terminal's cursor position would no longer match the renderer's model of it.

3. ESC_TIMEOUT. Raising it makes split sequences less likely but adds latency to every genuine Escape press, since Escape can only be confirmed by waiting.

4. It does not need to: the pen tracks the terminal's actual SGR state, which persists across line boundaries. Resetting per line would emit redundant escape sequences for no benefit. (Cursor position is a different matter and is re-emitted per run.)

Chapter 3

1. The redirect carries the authorization code but never the verifier. Exchanging a code requires presenting a verifier whose SHA-256 equals the challenge registered at the start — and only the legitimate client has it. The intercepted code is useless.

2. Each 401 triggers a refresh, which also fails, returning 401 — infinite recursion, hammering Spotify, no error ever surfaced. Retrying once distinguishes "stale cache" from "genuinely unauthorized".

3. Degraded dev-mode responses that omit uri while including id; without the fallback one malformed entry poisons a whole play payload. The risk is masking a genuinely unplayable item as playable — which is why play_tracks() independently filters malformed URIs before sending.

4. The GIL protects individual bytecodes, not multi-step sequences. The read–check–refresh–store sequence can interleave, so two threads could both refresh and one could invalidate the other's token.

Chapter 4

1. Comparisons and differences stay valid without modular arithmetic: _written - _readpos is the buffered amount directly. Concretely, recording a track boundary as an absolute position lets you later ask "is the cursor within X frames of it?" with plain subtraction — with wrapping indices, that test needs case analysis and breaks whenever the buffer wraps between marking and comparing.

2. Dropped frames are gaps: clicks, or a track that plays slightly fast while audio is silently discarded. Blocking instead converts a correctness problem into a scheduling one, and the sound card's real-time consumption resolves it.

3. After mixing, so a crossfade is visualized as heard rather than showing one track while two are audible. Before volume, so the bars do not shrink when you turn the volume down — the visualizer reflects the music, not the amplifier.

4. Audible almost immediately as static, since a one- or two-byte shift inverts and swaps channels. A test using an in-memory stream would likely miss it, because such reads return exactly the requested size — the bug needs a real pipe with partial reads, which is precisely why the codebase keeps an end-to-end pipe test.

Chapter 5

1. It carries intent. librespot emits it only when a track finishes naturally, never on a skip or seek, so it distinguishes cases that are acoustically identical. No stall-detection heuristic can tell "the track ended" from "the user skipped" — both look like a gap. It is also exact rather than probabilistic; the fast-cached-load case produces no stall to detect at all.

2. The boundary is computed too early by however much audio sits in the pipe, so the mix starts before the outgoing track has finished — you hear the next track fade in over the last moments of the current one, and the outgoing track's true ending is skipped.

3. Park change A while boundary B (from an earlier transition) is still queued. B plays. A boolean "has a boundary played" is now true, so A is adopted even though its own boundary has not been reached — the title flips early. The counter records the value at park time and requires a strict increase, so only a boundary that played after parking counts.

4. Flush at the moment of the keypress rather than in the response callback. The invariant needed at flush time is: the buffer contains only the outgoing track. That holds before Spotify has been told to switch, and stops holding as soon as librespot begins streaming the new one. Complementary fixes: keep the wide fill limit only near a natural boundary, and mark manual actions so a late end_of_track cannot attach a boundary to the new track.

Chapter 6

1. select waits on kernel file descriptors, and a Queue is user-space objects with a lock. Writing a byte to a pipe on completion gives the kernel something to signal, so the loop blocks until either input or a result arrives, with no timeout-based polling.

2. Data races on UI state. Callbacks run on the thread that owns the terminal and every widget, so no shared structure needs a lock and no half-updated frame can be rendered.

3. A poll requested before your keypress reflects the world before your action, even though it arrives after — that is precisely the response that must be discarded. Comparing arrival time would keep it (it arrived later) and clobber the optimistic update, reintroducing the flicker.

4. The global binding wins. _handle_key tests global keys in an elif chain and only reaches view.handle_key(self, key) in the final branch, so a global binding shadows every view's use of that character — the reason a new hotkey must be checked against all seven views first.

Chapter 7

1. (a) a field on Config — omit it and the attribute does not exist; (b) _KNOWN_KEYS — omit it and the value is dropped when config is loaded or saved, so the setting silently resets; (c) a Setting entry — omit it and the setting works but is unreachable from the UI.

2. Widgets read theme.X at render time, and the loop re-renders roughly 30 times a second. Rebinding the module attribute is therefore visible on the next frame. It exploits the fact that an immediate-mode renderer has no retained state to invalidate.

3. Terminal: Screen(out=..., size=...), which accepts a StringIO and a fixed size. Network: ZPOTIFY_CONFIG_DIR redirected to a temp directory plus monkeypatched API methods, so no credentials or sockets are involved.

4. A long-lived token is a bearer credential that exists somewhere — CI secrets, a developer machine, a shell history — and can be exfiltrated and replayed until revoked. OIDC trusted publishing issues a short-lived credential bound to a specific repository, workflow, and environment, so there is no durable secret to steal and a stolen one is not replayable elsewhere.
