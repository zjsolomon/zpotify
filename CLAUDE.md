# zpotify

Spotify in the terminal: a from-scratch TUI client. Python 3.12+, managed
with uv. Runtime deps are only `numpy` and `sounddevice` (PortAudio).
Audio comes from `librespot` (external Rust binary, found via PATH or the
Homebrew paths in `player/librespot.py`) run as a subprocess in pipe mode;
control goes through the Spotify Web API (stdlib `urllib`, hand-rolled OAuth
PKCE). Requires Spotify Premium.

## Commands

- `uv run pytest` — unit tests (fast, no network, no terminal needed)
- `uv run zpotify` — run the app (real terminal + librespot + Premium)
- `uv run zpotify doctor` — end-to-end health check; the best smoke test
- The dev machine's global `zpotify` is a `uv tool install --editable` shim,
  so it always runs this working tree.

## Architecture (src/zpotify/)

- `term/` — hand-rolled terminal engine: termios raw mode, ANSI diff
  renderer, key/mouse decoding. POSIX-only (termios, SIGWINCH): macOS and
  Linux work, Windows does not.
- `auth.py`, `api.py` — OAuth PKCE + Web API client, stdlib only.
- `player/` — librespot subprocess → PCM ring buffer → sounddevice output;
  FFT feeds the visualizer.
- `ui/` — event loop (`app.py`), seven tab views in `views/`, player bar.
- `ui/theme.py` — accent themes. Styles are module attributes read at render
  time; `theme.apply(name)` reassigns them, re-skinning the UI live. The
  default green look must stay pixel-identical (tests lock the gradient).
- Config lives in `~/.config/zpotify/` (override: `ZPOTIFY_CONFIG_DIR`).
  Adding a setting means: `Config` field + `_KNOWN_KEYS` in `config.py`,
  plus a `Setting` entry in `ui/views/settings.py`.

## README

- The ASCII UI mock in README.md is generated by
  `uv run python scripts/readme_mock.py` — regenerate it (and update the
  script) whenever the real layout changes; never let the diagram drift.
- The README also renders on PyPI: images and repo links must be absolute
  GitHub URLs (relative paths 404 there). The PyPI page only picks up README
  changes on the next release.

## Releasing to PyPI

Package name: `zpotify` (https://pypi.org/project/zpotify/). Publishing is
tokenless via GitHub Actions trusted publishing (OIDC):
`.github/workflows/release.yml` runs tests, then `uv publish
--trusted-publishing always` when a `v*` tag is pushed. The workflow uses the
`pypi` GitHub environment, and CI fails the release if the tag doesn't match
`pyproject.toml`'s version.

Release steps:
1. Bump `version` in pyproject.toml; update README/docs if user-facing.
2. Commit, then `git tag vX.Y.Z && git push origin main vX.Y.Z`.
3. CI publishes. Verify with `uvx --no-cache zpotify@X.Y.Z doctor` and the
   PyPI page (index propagation can lag a minute or two).

Manual fallback: `uv build && uv publish --token <project-scoped token>`.
Never commit tokens; nothing under `~/.config/zpotify/` (OAuth tokens,
librespot credentials) may ever enter this public repo.
